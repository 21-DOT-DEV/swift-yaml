## 1. Goal & success criteria

Decoding with the `.reject` setting throws a **positioned** error on the first duplicate **scalar** key in any mapping — block or inline, any depth — **of the first document**, while every other setting and the default behavior are unchanged. (Null, alias, and whole-structure keys are consumed for parity but not compared — see §2 and A2.)

**Concrete criteria:**
- `let d = YAMLDecoder(); d.duplicateKeyStrategy = .reject; try d.decode(T.self, from: "a: 1\na: 2")` throws, reporting key `a` at **line 2** (1-based). *(The strategy is a property, matching `JSONDecoder` — `YAMLDecoder` has only `init()`, no memberwise initializer.)*
- A duplicate nested in a block, or inside a flow `{a: 1, a: 2}`, is caught with its own position.
- Only the **first document** is checked — matching `YAMLDecoder`'s first-document decode scope; a duplicate confined to a later document of a stream is not this pass's concern (and decode discards those documents anyway).
- A document with no duplicate decodes exactly as before; the default (`.useLast`) and `.useFirst` are unchanged.
- The size and depth safety budgets still apply under `.reject` (the pass runs *after* them).

## 2. Scope

**In scope:** the `.reject` case; the shim `EventHandler` + `yamlx::firstDuplicateKey`; the decode-path pre-pass wiring; the `duplicateKey` error; tests; the doc corrections.

**Out of scope (deferred owners):** a **standalone check-only function** (add later if a validate-without-decoding need is confirmed — trivial over the same detector); **collect-all reporting** (the linter shape — belongs to that future function); **checking documents past the first** in a stream (the pass matches decode's first-document scope; a later-document check belongs to a future decode-all/standalone path); **dup-checking a null, alias, or whole-structure key** (rare — consumed for parity, not compared; documented gaps); **making `.useFirst` faithful** (the handler enables it, but that's a separate follow-on); the **pure-Swift text-scan** alternative (rejected in D1).

## 3. Design / approach

**The handler** — C++, declared in **`Sources/yamlcppShims/include/yamlcpp_shims.h`** (the target exposes only `include/`, so a non-`include/` file would be invisible to Swift — F6). It needs an explicit **`#include "yaml-cpp/eventhandler.h"`** (the umbrella `yaml.h` only *forward-declares* `EventHandler`, `parser.h:17`), plus `<set>`, `<vector>`, `<sstream>`, `<string>` for the state and input stream (F5). A `YAML::EventHandler` subclass **overriding all nine pure-virtuals with their exact signatures** (F7) — note the `*End` events take no `Mark`, and the `*Start` events take `tag`, `anchor`, and `style` besides the `Mark`:
`OnDocumentStart(const Mark&)`, `OnDocumentEnd()`, `OnNull(const Mark&, anchor_t)`, `OnAlias(const Mark&, anchor_t)`, `OnScalar(const Mark&, const std::string& tag, anchor_t, const std::string& value)`, `OnSequenceStart(const Mark&, const std::string& tag, anchor_t, EmitterStyle::value)`, `OnSequenceEnd()`, `OnMapStart(const Mark&, const std::string& tag, anchor_t, EmitterStyle::value)`, `OnMapEnd()`. `OnDocumentStart`/`OnDocumentEnd` are no-ops that reset per-document state; the rest drive detection. State is a stack of frames — a *map* frame carries a `std::set<std::string>` of seen key texts and an `expectingKey` bool; a *sequence* frame is a marker.

**Key/value parity** (F2) — mirror yaml-cpp's own `NodeBuilder`, which tracks key-vs-value with a stack (`m_keys`, `nodebuilder.cpp:84-122`). Every *node event* first **consumes one slot** in the enclosing map: if the top frame is a map, the incoming node is the **key** when `expectingKey` (then set `expectingKey = false`) or the **value** otherwise (then set `expectingKey = true`); a sequence top (or empty stack) is a no-op. Because a nested collection consumes its slot at its **start**, the parent's `expectingKey` is already correct when it pops — so ends need no parent update, and a map-valued key, a map-valued value, and a complex key all stay in parity.
- `OnScalar(mark, tag, anchor, value)`: note whether this is a key slot, consume the slot, then **if it was a key slot** → dup-check `value` (first repeat → record `mark` + `value`, set `found`, stop).
- `OnNull(mark, anchor)` / `OnAlias(mark, anchor)` (F3): consume the slot for parity, but **do not compare** — these signatures carry no `value`, so a null or alias key has no comparable text; null/alias keys are a documented gap (A2).
- `OnMapStart` / `OnSequenceStart`: consume the slot, then push a frame. `OnMapEnd` / `OnSequenceEnd`: pop.

Traced against `a:\n  b: 1\n  b: 2` it flags the nested `b` at its mark; a complex (structure) key consumes its slot but isn't compared (the gap).

**The shim function** (F1, F4, F6). `inline DupResult firstDuplicateKey(const char *input)` in `yamlcpp_shims.h`, returning `struct DupResult { bool found; long line; long column; CStr key; }`. It wraps `input` in a **`std::istringstream`** and constructs `YAML::Parser(stream)` — the parser has no `const char*` constructor (F1) — then calls `HandleNextDocument(handler)` **once, on the first document only** (A2: matches `YAMLDecoder`'s first-document scope; a later document a stream carries but decode ignores is not this pass's concern). The duplicate key text is copied into a **`static thread_local std::string`** that outlives the return, and `DupResult.key` is a `CStr` view of *that* buffer (F4 — mirrors `scalarText()`/`parseMessage()`; a view of a local `std::string` would dangle once the function returns). `line`/`column` are the raw **0-based** `Mark` values. Exception-guarded (malformed input → `found = false`; the normal load surfaces the parse error).

**Decode integration** (F4, F5, F9). Inside `YAMLSerialization.parse`, **after** the existing `maxInputBytes` and `checkNesting` guards run (F5 — the `.reject` pass must never bypass the size/depth budgets on untrusted input; defense-in-depth) and **only when** `duplicateKeyStrategy == .reject`: run `firstDuplicateKey`; if `found`, build `YAMLError.duplicateKey(key:, line:, column:)` with `line`/`column` **converted `+1`** to 1-based (F9 — `Mark` is 0-based), then **throw `DecodingError.dataCorrupted(Context(codingPath: [], debugDescription: err.description, underlyingError: err))`** — mirroring the `.parse` path *exactly* (F4), so a bare `YAMLError` never escapes to `catch DecodingError` callers. Every other strategy skips the pass (zero added cost).

**Error.** New `YAMLError.duplicateKey(key: String, line: Int, column: Int)`, position **1-based — the overlay converts yaml-cpp's 0-based marks** (the accurate wording; *not* "as reported by yaml-cpp", the misleading phrasing the pre-existing `.parse` doc and `Decoding.md` use — corrected too, §6). Adding this case **requires a matching branch in `YAMLError.description`'s exhaustive switch** (F3 — otherwise it won't compile). It is a source-level addition to the public `YAMLError` enum; callers catch `DecodingError` and read `.description` (as the library's own docs describe) rather than exhaustively switching, so the practical impact is nil — an exhaustive-switch client would need `@unknown default`. The enum case is **`.reject`** — *not* `.throw`, a Swift reserved word that would force back-ticks (F8; best practice is a non-keyword name; the roadmap, which said `.throw`, is corrected to match).

## 4. Tests (`Tests/YAMLTests/DuplicateKeyTests.swift`)

- **Rejects (with position):** top-level `a: 1\na: 2`; nested; inline flow `{a: 1, a: 2}`; a repeated `<<`.
- **Accepts / unchanged:** no-duplicate documents decode unchanged; sibling keys at different levels or in different mappings are fine; the same key text in *different* mappings is fine; **a duplicate confined to a *later* document of a stream is not rejected** (the pass, like decode, reads only the first document).
- **Unchanged:** `.useLast` (default) and `.useFirst` behavior identical to today on a repeated key.
- **Documented edges:** `1:` + `"1":` is rejected (over-catch, by design); a whole-structure key is not dup-checked.
- **Oracle:** the position is asserted against the known fixture line/column.

## 5. Risks & mitigations

- **Handler state-machine correctness** (key/value toggle across nested maps/sequences, aliases, null values). *Mitigation:* the opening spike + spec-derived and hand-authored fixtures; the stack model is the standard event-to-structure machine.
- **Tag / complex-key / merge-key edges.** *Mitigation:* scoped out and documented (A2); by-text equality (D4) sidesteps tag-resolution entirely.
- **Extra pass overhead.** *Mitigation:* opt-in (`.reject` only) and tree-free (cheaper than a load); acceptable for a validation setting.
- **yaml-cpp exception safety.** *Mitigation:* guard the shim function as the others are (catch, return `found = false`; malformed input still surfaces via the normal load's parse error).
