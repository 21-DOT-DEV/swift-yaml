## 1. Goal & success criteria

Let a caller delete one entry from a hand-maintained YAML document and get back text whose only change is that entry's line — comments, ordering, and formatting everywhere else byte-identical, and the result still parses.

**Roadmap success metrics (L40–42):**
- Removing a key leaves sibling keys, blank lines, and comments unchanged; the document still parses.

**Concrete criteria:**
- `try YAMLEditor.unset(yaml, at: ["server", "port"])` removes the whole `port:` line (and its trailing comment); every other line — including a comment on the line above — is byte-identical; the result re-parses without `server.port`.
- `try YAMLEditor.unset(yaml, at: ["features", 0])` removes the first list item's line; the remaining items shift down by one.
- Deleting the only entry of a map/list leaves an empty (null) parent that still parses.
- A missing path, a nested-section target, a multi-line/flow value, or an anchor/alias document returns a typed `YAMLEditError`, never a silent no-op.

## 2. Scope

**In scope:** `YAMLEditor.unset` (static); deletion of a **leaf map key** or **leaf list item** whose value is a single-line scalar; line-range removal; reuse of feature 1's navigation, span check, guards, and error taxonomy; tests.

**Out of scope (typed errors / deferred owners):** a key/element whose value is a **nested section** — a block *or* inline/flow map/sequence (`key: {…}` / `key: [a, b]`) → `notASingleValue` (whole-block deletion is a later increment — it needs indentation-extent logic feature 1 doesn't have); a **scalar inside an inline (flow) collection** (`key: [a, b]` at `["key", 0]`) → `unsupportedValueShape` (it shares its line with the parent key and siblings, so a line delete would wipe the whole collection — the flow-context guard, §3 step 5); a **multi-line/block** scalar value → `unsupportedValueShape`; an **empty (null)** value → `unsupportedValueShape` (consistent with feature 1; a later increment); a value on a **continuation line** (`key:` / `-` then the value indented below) → `unsupportedValueShape` (its key/marker is on a different line, so a single-line delete would leave a bare `key:`/`-`; whole-entry cross-line deletion is a later increment); a **compact block-sequence entry** — a map key sharing its line with a `-` list marker (`- key: value`), or a nested `- - …` line → `unsupportedValueShape` (a whole-line delete would drop an *enclosing* list marker and corrupt the document; deleting the whole item is the `.index` path — a map item is `notASingleValue` — and compact/cross-line deletion is a later increment); **multi-document** input → first document only; **anchor/alias** documents → `documentUsesAnchorsOrAliases`; deleting a **leading comment block** with the key (D2); collapsing blank lines (A4).

## 3. Design

**Public surface** (added to the existing `YAMLEditor`):
```swift
extension YAMLEditor {
    /// Deletes the leaf entry at `path` (a map key or a list item) from `yaml`,
    /// returning the edited text. The entry's whole line — key/`-`, value, and any
    /// trailing same-line comment — is removed; everything else is byte-identical.
    /// Deleting a list item shifts later items' positions down by one.
    public static func unset(_ yaml: String, at path: [YAMLPath.Component]) throws -> String
}
```

**Pipeline** (all Swift; `bytes = Array(yaml.utf8)` throughout, indices align with `valueSpan` — as in feature 1):
1. **Refuse anchors/aliases** — reuse `documentUsesAnchorsOrAliases(bytes)`; if it fires → `documentUsesAnchorsOrAliases`.
2. **Parse, guarded** — reuse feature 1's step (`yamlx.parse`; on `!ok` → `malformedDocument(.parse(…))`).
3. **Navigate** — reuse feature 1's validated walk (`.key` → require map, `mapChild`; `.index` → require sequence + in-range, `seqItem`; undefined child → `pathNotFound(path:)`).
4. **Confirm a single-line leaf.** `kind(node)`: a map/sequence → `notASingleValue(path:)`; null → `unsupportedValueShape(path:)`. A scalar → `let span = yamlx.valueSpan(node, cstr, bytes.count)`; guard `span.ok`; confirm single-line: `span.endReliable` (plain), else a **same-line** quoted-end scan succeeds (`bytes[pos]` is `"`/`'` and the close is found before a newline), else → `unsupportedValueShape(path:)`.
5. **Find the entry's line** from `pos = Int(span.pos)`:
   - `lineStart` = smallest `i ≤ pos` with `i == 0` or `bytes[i-1] == newline` (the existing `newline = UInt8(ascii: "\n")` constant — `bytes` is `[UInt8]`, so compare bytes, never a `String` literal).
   - **Same-line guard:** `bytes[lineStart..<pos]` must hold a non-whitespace byte (the key+`:` or the `-`). If it is all spaces/tabs, the value is on a continuation line separate from its entry marker → `unsupportedValueShape(path:)` (deleting only this line would orphan a bare `key:`/`-`). Exact: a same-line entry always has a non-whitespace marker before the value (no false positives), and a continuation value is always alone on its line (no false negatives).
   - **Enclosing-marker guard:** the target's line may carry at most the target's *own* block-sequence marker — a single leading `- ` when the final path component is a list index, and **none** when it is a map key. Count only the **leading block-marker run**: from `lineStart`, skip the indent, then count consecutive `- ` (dash + space/tab) markers, stopping at the first non-marker byte (where the key/scalar begins); if the count exceeds the allowance → `unsupportedValueShape(path:)`. Counting *only the leading run* — not every `- ` between `lineStart` and `pos` — is essential: a `-` *inside* a key, quoted (`"a - b"`) or plain (`a - b`), is scalar content, not a structural marker (YAML indicators are literal inside/within a scalar), so it must not be counted, or a valid key deletion would be wrongly rejected. This catches a **compact block sequence**: `- name: alpha` (deleting the `.key` `name` would drop the `-` that belongs to the *list item*, merging its mapping into the parent — corrupt output `libyaml`/Ruby reject) and a nested `- - alpha` (deleting the inner `.index` item would drop the outer `-`). yaml-cpp writes and accepts this compact form (the item's mapping starts on the `-` line) and the value node's mark is on that shared line, so nothing downstream flags it.
   - **Flow-context guard:** track flow-bracket **depth from the document start** to `pos`, **quote/comment-aware**; if depth > 0 at `pos`, the target is a scalar *inside* an inline (flow) collection (`features: [alpha, beta]` at `["features", 0]`, or `config: {host: x}` at `["config", "host"]`) — which shares its line with the parent key and siblings, so a line delete would wipe the collection (and for a **multi-line** flow collection like `data: [x,\ny, z]`, strip the closing bracket, silently corrupting). Reject → `unsupportedValueShape(path:)`. Depth-from-start (not a line-only scan) is required because a flow opener can sit on an **earlier line** — flow collections span lines. A `[`/`{` inside a quoted key (`"a[b]"`), inside a comment, or in a balanced `[…]` that closed before the target, does not count. This guard is essential because yaml-cpp doesn't distinguish flow context — the inner scalar is `kindScalar` and `valueSpan`-reliable, so it slips past classify and the same-line/enclosing-marker guards.
   - `lineEnd` = smallest `j > pos` with `bytes[j-1] == newline`, or `bytes.count` (last line, no trailing newline).
6. **Delete the line:** `return String(decoding: bytes[0..<lineStart] + bytes[lineEnd...], as: UTF8.self)`.

**Why line-based is correct here.** In a block mapping/sequence there is exactly one entry per line, so removing `[lineStart, lineEnd)` removes the whole entry — indent, key/`-`, value, and any trailing comment — and nothing else. Scanning back from the *value's* position captures the key (which precedes it on the line) **when the same-line guard holds**, so no key-node accessor is needed. Two checks keep it safe: the single-line-*value* check (step 4) rejects a value that itself spans lines, and the same-line-*entry* guard (step 5) rejects a value sitting on a line separate from its `key:`/`-` — the case that would otherwise leave a bare marker.

## 4. Tests (`Tests/YAMLTests/`, extending `EditorTests`)

Spec-grounded fixtures (YAML 1.2.2 numbered examples, the corpus already cited), asserted with the re-parse oracle:
1. **Comment/sibling/blank preservation (Example 2.2).** Delete `avg`; assert the `hr` and `rbi` lines — *with their `#` comments* — are byte-identical, the `avg` line (and its `# Batting average`) is gone, and the result re-parses without `avg`.
2. **Own-line comment goes, other comments stay.** A fixture with a leading comment above the target and a trailing comment on it: after `unset`, the trailing comment is gone with the line, the leading comment remains verbatim.
3. **List item + index shift.** Delete `["features", 0]` from a block sequence; the first item's line is gone and the old second item is now first (oracle at index 0).
4. **Delete-only-entry still parses.** Deleting the sole child of a map (or sole list item) leaves an empty parent that re-parses.
5. **Blank lines untouched.** A target flanked by blank lines: after deletion the blank lines are unchanged (no collapse).
6. **Errors** — `pathNotFound` (bad key / out-of-range index), `notASingleValue` (a nested-section target), `unsupportedValueShape` (multi-line/block value, null value), `documentUsesAnchorsOrAliases` (Example 2.10).
7. **Continuation-line value → error, never a bare key.** `k:\n  v` and a list `-\n  v` (value on the line below its key/marker) return `unsupportedValueShape`; assert the result is *not* a `k:`/`-`-with-null mis-deletion.
8. **Compact block sequence → error, never corruption.** A list-of-maps `features:\n  - name: alpha\n    port: 8080` with target `["features", 0, "name"]` (the first key, sharing the `-` line) returns `unsupportedValueShape`; a nested `- - alpha` inner-item target likewise. Deleting the *second* key `["features", 0, "port"]` (on its own line, no `-`) **is** allowed — asserted to succeed and re-parse.
9. **Flow-collection scalar → error, never over-deletion.** `features: [alpha, beta]` at `["features", 0]`, and `config: {host: localhost}` at `["config", "host"]`, return `unsupportedValueShape`; assert the source is unchanged (the whole line is *not* wiped). A quoted key with a bracket (`"a[b]": v`) is still deletable (the `[` is inside quotes).

## 5. Risks & mitigations

- **Half-deleting a multi-line value.** If a key's value spanned several lines and we removed only the first, the rest would be orphaned. *Mitigation:* the single-line-scalar check (step 4) rejects block/multi-line/nested targets before any deletion; tested.
- **Bare key/marker on a continuation-line value (review round 1).** For a valid block entry `key:\n  value` (or `-\n  value`) the value node's mark is on the *value's* line, not the key's (verified `singledocparser.cpp:256-266`, `nodebuilder.cpp:76-82`), and the single-line-value check passes — so a naive line-delete would remove only the value line and leave a null `key:`/`-`, a *silent* mis-deletion. *Mitigation:* the same-line guard (step 5) rejects these with `unsupportedValueShape`; tested. Whole-entry deletion across the key line, the value line, and any comments between is a deferred increment (genuinely harder: robustly finding the key line, preserving interleaved comments, and list `-` markers have no node position).
- **Corrupting a compact block sequence (review round 2).** In compact notation a list item's mapping starts on the `-` line (`- name: alpha`), so a `.key` target's line also carries the item's `-`; a whole-line delete drops that marker and merges the mapping into its parent (`libyaml`/Ruby then reject the result). A nested `- - alpha` line carries two markers. *Mitigation:* the enclosing-marker guard (step 5) rejects these with `unsupportedValueShape`; tested. Deleting a *whole* compact item is the `.index` path — and a map item is `notASingleValue`, already deferred.
- **Over-deleting a flow collection (review rounds 4 & 5).** A scalar inside an inline collection is `kindScalar`, single-line, and `valueSpan`-reliable — yaml-cpp doesn't track flow context — so it passes classify and the single-line/same-line/enclosing-marker guards; but it shares its line with the parent key and siblings, so a line delete would wipe the collection (`features: [alpha, beta]`), and for a **multi-line** flow (`data: [x,\ny, z]`) strip the closing bracket, *silently corrupting*. *Mitigation:* the flow-context guard (step 5) tracks bracket **depth from the document start** (not just the target's line) and rejects a positive depth with `unsupportedValueShape`; tested (single- and multi-line, `[…]` and `{…}`). Editing *within* a flow collection needs intra-line splicing, a later increment.
- **Line endings.** The line-boundary scan handles **CRLF** (Windows) — the `\r` is part of the line and the delete is byte-exact (tested). A lone-`\r` (old-Mac) document is **rejected by yaml-cpp at parse** — it does not treat a lone `\r` as a line break (verified: "illegal map value") — so `unset` throws `malformedDocument` and never reaches line deletion; the "one giant line → total wipe" hazard therefore cannot occur (tested by asserting the throw). The scan also *defensively* treats a lone `\r` as a break, so it stays safe even if that parse behavior ever changed.
- **Block scalar with unbalanced brackets (safe, fail-closed).** The flow-context guard scans source bytes and does not skip block-scalar (`|`/`>`) bodies, so an unbalanced `[`/`{` in a *sibling* block scalar before the target would inflate the depth and *refuse* an otherwise-valid deletion. It's a refusal, never corruption, and rare (block-scalar content usually has balanced or no brackets). A future robustness pass could read yaml-cpp's collection `Style()` (via a small shim) to detect flow context directly, retiring this edge and the source-scan fragility both.
- **First/last line & no-trailing-newline edges.** *Mitigation:* `lineStart`/`lineEnd` handle `pos` on the first line (`lineStart == 0`) and the last line (`lineEnd == bytes.count`); tested on both.
- **Comment misattribution (D2).** Deliberately avoided — we never delete a comment that isn't on the entry's own line, so we can't remove a section header by mistake. The cost (a leading note may be left orphaned) is documented.
- **Multibyte/CRLF alignment.** Same basis as feature 1 — `valueSpan` positions index the same UTF-8 bytes; a byte-order-mark skew makes the single-line check fail closed (`unsupportedValueShape`).
